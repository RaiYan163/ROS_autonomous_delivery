MCP + ROS 2 + TurtleBot3 — project notes
========================================
Workspace: mcp_approach (ROS 2 Jazzy; TurtleBot3 sim with waffle_pi for camera)

1) What we set out to do
------------------------
- Minimal MCP server that integrates with ROS 2.
- First milestone: publish std_msgs/String to a topic from MCP, with an OpenAI-based
  client that routes user text to tool calls.
- Planned next: drive TurtleBot3 in simulation (/cmd_vel) and expose live camera
  (and optionally lidar) through MCP tools.

2) ROS 2 topics you care about (waffle_pi in Gazebo)
----------------------------------------------------
- /chatter              — demo String topic (early MCP publish_message test).
- /cmd_vel              — geometry_msgs/TwistStamped in Gazebo (ros_gz bridge); MCP
  teleop matches that type.
- /scan                 — sensor_msgs/LaserScan; lidar.
- /camera/image_raw     — sensor_msgs/Image; main camera (~30 Hz in your tests).
- /camera/camera_info   — sensor_msgs/CameraInfo; intrinsics/metadata.

Note: default burger often has lidar but no camera in this stack; waffle_pi gives
camera + lidar for sim.

3) Issues we ran into (and what fixed them)
-------------------------------------------
a) pip "externally-managed-environment" (PEP 668 on Debian/Ubuntu)
   - Use a dedicated venv or conda env; use that env's python -m pip.

b) conda shell showed (mcp_turtle) but which python / which pip were wrong
   - Use python -m pip; ensure conda env actually provides python.

c) ROS Jazzy expects Python 3.12 for /opt/ros/jazzy/.../site-packages/rclpy
   - Use Python 3.12 (e.g. /usr/bin/python3.12 or conda env python=3.12).

d) import rclpy failed: No module named 'yaml'
   - Install PyYAML in the same interpreter (pip package pyyaml).

e) create_node failed deeper in stack: No module named 'numpy'
   - Install numpy in that interpreter.

f) mcp_server/mcp_client_openai.py: server subprocess could not import rclpy while interactive
   shell could — environment not passed to child process.
   - Fix: StdioServerParameters(..., env=os.environ.copy()) so PYTHONPATH and ROS
     vars reach the server.

g) Always source ROS in the shell before running rclpy-based code:
   source /opt/ros/jazzy/setup.bash
   After colcon build: source install/setup.bash

4) Viewing the camera in real time
----------------------------------
- Confirm stream:
  ros2 topic hz /camera/image_raw

- Viewer (ROS 2 invocation; bare `rqt_image_view` may not be on PATH):
  ros2 run rqt_image_view rqt_image_view
  ros2 run rqt_image_view rqt_image_view /camera/image_raw

- Install if missing:
  sudo apt install ros-jazzy-rqt-image-view

- Grey/blank rqt_image_view: use topic dropdown → pick /camera/image_raw (refresh if
  list is empty). If hz shows ~30 Hz, data is publishing.

7) SLAM: Gazebo, then Cartographer (RViz), time sync, save map
--------------------------------------------------------------
Order matters. Use the same TURTLEBOT3_MODEL (e.g. waffle_pi) in every terminal.

Terminal 1 — Gazebo world (start first; wait until robot is spawned)
  cd ~/Desktop/mcp_approach
  source /opt/ros/jazzy/setup.bash
  source install/setup.bash
  export TURTLEBOT3_MODEL=waffle_pi
  ros2 launch turtlebot3_gazebo turtlebot3_world.launch.py

Terminal 2 — Cartographer (starts cartographer_node + occupancy grid + RViz)
  cd ~/Desktop/mcp_approach
  source /opt/ros/jazzy/setup.bash
  source install/setup.bash
  export TURTLEBOT3_MODEL=waffle_pi
  ros2 launch turtlebot3_cartographer cartographer.launch.py use_sim_time:=true

Why use_sim_time:=true (symptom if you skip it)
  Gazebo publishes /clock (simulation time). The world launch uses use_sim_time on
  robot_state_publisher and bridges, so /tf and /scan are stamped with sim time.
  turtlebot3_cartographer defaults use_sim_time to false, so RViz and Cartographer
  use wall clock → TF lookups fail, RViz shows "No transform from base_scan (etc.)
  to map", MessageFilter drops laser messages, map looks wrong. Fix: always pass
  use_sim_time:=true when mapping in sim.

Quick check after launch
  ros2 param get /rviz2 use_sim_time
  ros2 param get /cartographer_node use_sim_time
  (both should be true when using Gazebo)

Terminal 3 — Keyboard teleop (optional; only while driving)
  source /opt/ros/jazzy/setup.bash
  source ~/Desktop/mcp_approach/install/setup.bash
  export TURTLEBOT3_MODEL=waffle_pi
  ros2 run turtlebot3_teleop teleop_keyboard
  Install if needed: sudo apt install ros-jazzy-turtlebot3-teleop
  If the robot does not move, the sim bridge expects TwistStamped on /cmd_vel; use
  MCP teleop_move or a compatible publisher.

Save the occupancy map (after exploring; keep Gazebo + Cartographer running)
  source /opt/ros/jazzy/setup.bash
  sudo apt install ros-jazzy-nav2-map-server   # once
  mkdir -p ~/Desktop/mcp_approach/maps
  cd ~/Desktop/mcp_approach/maps
  ros2 run nav2_map_server map_saver_cli -f my_map_name
  Produces my_map_name.yaml and my_map_name.pgm (path is prefix, no extension).
  Sanity: ros2 topic echo /map --once

Clean shutdown / restart
  Stop each terminal with Ctrl+C, or run ./kill.sh from the repo root (if present),
  then start again from Terminal 1.

8) MCP tools implemented (TurtleBot bridge)
------------------------------------------
Run the client (Gazebo should already be up; same machine as the sim):

  cd ~/Desktop/mcp_approach
  source /opt/ros/jazzy/setup.bash
  source install/setup.bash
  export TURTLEBOT3_MODEL=waffle_pi
  export MCP_USE_SIM_TIME=1          # match Gazebo /clock so cmd_vel time is consistent
  pip install -r requirements.txt    # if needed: mcp, openai, dotenv, pyyaml, numpy, opencv
  # .env must define OPENAI_API_KEY (and optional OPENAI_MODEL)
  python3 -m mcp_server.mcp_client_openai

Then type natural commands, e.g. "drive forward for 3 seconds", "turn left 2s", "stop".
The client routes to tool teleop_move (TwistStamped on /cmd_vel).

- publish_message(text) -> /chatter
- get_status() -> lists chatter, cmd_vel, odom, camera topics and snapshot dir
- get_camera_snapshot() -> saves latest /camera/image_raw to temp/mcp_snapshots/*.jpg
- teleop_move(direction, duration_sec) -> /cmd_vel for duration then stop
  (direction: forward | back | left | right | stop)
- get_odometry() -> summary from /odom (pose + yaw + twist)
- get_navigation_help() -> Nav2 workflow (initial pose, goal, execute)
- set_navigation_initial_pose(x, y, yaw_deg) -> publish /initialpose (AMCL, map frame)
- set_navigation_goal(x, y, yaw_deg=0) -> store goal for execute_navigation
- get_navigation_state() -> show stored initial pose and goal
- execute_navigation() -> NavigateToPose action for stored goal (Nav2 must be running)

Snapshot dir: temp/mcp_snapshots/ (under project root). Install OpenCV for snapshots:
  pip install opencv-python-headless

10) MCP Nav2 navigation — step-by-step (sequential)
--------------------------------------------------
Prereqs (once per machine)
- Source ROS and workspace: source /opt/ros/jazzy/setup.bash && source install/setup.bash
- pip install -r requirements.txt (mcp, openai, etc.)
- .env in repo root: OPENAI_API_KEY (and optional OPENAI_MODEL) for mcp_server.mcp_client_openai
- Use the same TURTLEBOT3_MODEL (e.g. burger or waffle_pi) in every terminal that runs
  Gazebo / Nav2 / MCP.

How the MCP server runs (important)
- Default workflow: you do NOT start the server in a separate terminal by hand.
  mcp_server/mcp_client_openai.py launches mcp_server/mcp_server_ros2.py automatically as a subprocess (stdio
  MCP) and passes env=os.environ.copy() so ROS and PYTHONPATH work in the server.
- To run only the server (e.g. Cursor MCP config, debugging, another MCP host):
    cd ~/Desktop/mcp_approach
    source /opt/ros/jazzy/setup.bash
    source install/setup.bash
    export TURTLEBOT3_MODEL=burger
    python3 mcp_server/mcp_server_ros2.py
  That speaks MCP over stdio; it blocks the terminal until you stop it.

Terminal layout (recommended)
- Terminal 1: Gazebo
- Terminal 2: Nav2 + RViz
- Terminal 3: OpenAI chat client (starts MCP server for you — see below)

1) Gazebo
  cd ~/Desktop/mcp_approach
  source /opt/ros/jazzy/setup.bash
  source install/setup.bash
  export TURTLEBOT3_MODEL=burger
  ros2 launch turtlebot3_gazebo turtlebot3_world.launch.py
  Wait until the robot is spawned.

2) TurtleBot3 Nav2 (required for navigation tools)
  cd ~/Desktop/mcp_approach
  source /opt/ros/jazzy/setup.bash
  source install/setup.bash
  export TURTLEBOT3_MODEL=burger
  ros2 launch turtlebot3_navigation2 navigation2.launch.py use_sim_time:=true \
    map:=$PWD/map/turtlebot3_world_explored.yaml
  In RViz: 2D Pose Estimate so laser aligns with map (or set initial pose later via MCP).

3) Environment for MCP (same shell you will use for the client)
  cd ~/Desktop/mcp_approach
  source /opt/ros/jazzy/setup.bash
  source install/setup.bash
  export TURTLEBOT3_MODEL=burger
  export MCP_USE_SIM_TIME=1          # recommended with Gazebo (matches /clock)
  # Ensure .env exists with OPENAI_API_KEY

4) Start MCP: run the OpenAI client (this starts the MCP server automatically)
  python3 -m mcp_server.mcp_client_openai
  You should see a prompt like "OpenAI MCP client ready". Type commands at "You:".
  Behind the scenes: child process = python3 mcp_server/mcp_server_ros2.py (do not run it again
  in another terminal unless you are using a different integration).

5) Chat — natural language routes to tools
  Examples:
  - "How do I navigate?" -> get_navigation_help
  - Set initial pose (map frame, meters, degrees) for AMCL
  - Set navigation goal (x, y, yaw_deg) — stored until execute
  - "Execute navigation" / "go to the goal" -> execute_navigation (NavigateToPose)

6) Minimal navigation sequence in chat
  1) Ask for navigation help if needed.
  2) Set initial pose (x, y, yaw_deg) matching where the robot is in Gazebo.
  3) Set navigation goal (x, y, yaw_deg).
  4) Execute navigation.

Tips
- Nav2 must be running before execute_navigation; /navigate_to_pose must exist.
- Do not use teleop_move while Nav2 is driving (both use /cmd_vel).
- execute_navigation sends only NavigateToPose; it does not re-publish initial pose
  (localize first via RViz or set_navigation_initial_pose).
- You can change initial pose or goal anytime by calling the setters again.
